## 2. Fix Files Disappearing on Refresh

### Problem
The `refreshFilesInternal` clears or deletes local files based on potentially incomplete data from Telegram during early sync or due to TDLib's 100-message limit. This causes a temporary disappearance of files until a later refresh repopulates them.

### Solution Strategy
- Skip all deletion logic during the **first refresh** after app start (when TDLib may still be warming up).
- Only delete missing files when we are sure we have retrieved the complete set (i.e., the returned batch size is less than the request limit of 100). This avoids deleting older files that exist in the DB but were not returned due to pagination limits.

### 2.1 Modify DriveViewModel
**File**: `app/src/main/java/com/teledrive/android/ui/drive/DriveViewModel.kt`
- Add constants:
  ```kotlin
  private const val LIST_FILES_LIMIT = 100
  ```
- Add state:
  ```kotlin
  private var isFirstRefresh = true
  ```
- Refactor `refreshFilesInternal`:
  ```kotlin
  private suspend fun refreshFilesInternal(folderId: Long?) {
      val gatewayFiles = gateway.listFiles(folderId)
      val files = gatewayFiles.map { it.toEntity() }

      // If the gateway returns an empty list, only clear if this is NOT the first refresh.
      if (files.isEmpty()) {
          if (!isFirstRefresh) {
              dao.clearFilesForFolder(folderId)
          }
          // Mark first refresh complete and return.
          isFirstRefresh = false
          return
      }

      dao.upsertFiles(files)

      // Build preview cache entries (may have null thumbnail path initially)
      val previewEntries = gatewayFiles.mapNotNull { tf ->
          val key = previewCacheKeyFor(tf.messageId, tf.folderId, tf.tdRemoteUniqueId)
          val thumbPath = tf.tdThumbnailLocalPath?.takeIf { java.io.File(it).exists() }
          PreviewCacheEntity(
              cacheKey = key,
              thumbnailLocalPath = thumbPath,
              thumbnailReady = thumbPath != null,
              previewKind = if (thumbPath != null) "thumbnail" else "badge",
              badgeExt = tf.extension,
              mimeType = tf.mimeType,
              sourceFileName = tf.name,
              updatedAt = System.currentTimeMillis(),
          )
      }
      if (previewEntries.isNotEmpty()) dao.upsertPreviewCache(previewEntries)

      // Only delete files that are missing from this result if we likely have the complete list.
      // Complete when returned size < limit OR we are past the first refresh and the size is less than limit.
      // We never delete when size == limit because there may be more files beyond the limit.
      if (!isFirstRefresh && files.size < LIST_FILES_LIMIT) {
          val messageIds = files.map { it.messageId }
          if (folderId == null) dao.deleteMissingRootFiles(messageIds) else dao.deleteMissingFilesInFolder(folderId, messageIds)
      }

      // First refresh completed
      isFirstRefresh = false

      // Kick off asynchronous thumbnail downloads (see Section 3)
      scheduleThumbnailDownloads()
  }
  ```
- Remove the early `return` after deletion, ensure `isFirstRefresh` is always updated.

## 3. Thumbnail Reliability & Async Loading

### Issues
- `listFiles` synchronously waits for each thumbnail (up to 4s) causing slow loads and timeouts → some videos lack thumbnails.
- Thumbnail download completion never persisted to DB.

### Solution
- **Remove** synchronous thumbnail prefetch from `listFiles`.
- Add explicit asynchronous thumbnail download in `DriveViewModel`.
- Persist downloaded thumbnail paths to DB and preview cache.

### 3.1 Extend TelegramGateway interface

### 3.4 Asynchronous thumbnail scheduling in DriveViewModel
**File**: `app/src/main/java/com/teledrive/android/ui/drive/DriveViewModel.kt`
- Add state:
  ```kotlin
  private val pendingThumbnailDownloads = mutableSetOf<Long>() // messageIds
  ```
- Add DAO helper method (or direct query):
  - In `TeleDriveDao`, add:
    ```kotlin
    @Query("SELECT * FROM files WHERE messageId = :messageId")
    suspend fun getFileByMessageId(messageId: Long): FileEntity?
    ```
- Implement `scheduleThumbnailDownloads()`:
  ```kotlin
  private fun scheduleThumbnailDownloads() {
      val files = uiState.value.files
      val needing = files.filter {
          it.tdThumbnailFileId != null &&
          it.tdThumbnailLocalPath.isNullOrBlank() &&
          it.messageId !in pendingThumbnailDownloads
      }
      if (needing.isEmpty()) return

      viewModelScope.launch {
          for (file in needing) {
              // Acquire semaphore to limit concurrent downloads
              thumbnailSemaphore.withPermit {
                  if (file.messageId in pendingThumbnailDownloads) return@withPermit
                  pendingThumbnailDownloads += file.messageId
                  try {
                      val path = gateway.downloadThumbnail(file.tdThumbnailFileId!!)
                      if (path != null) {
                          // Update file entity
                          val updated = file.copy(tdThumbnailLocalPath = path)
                          dao.upsertFiles(listOf(updated))
                          // Upsert preview cache entry
                          val cacheKey = previewCacheKeyFor(file.messageId, file.folderId, file.tdRemoteUniqueId)
                          dao.upsertPreviewCache(
                              listOf(
                                  PreviewCacheEntity(
                                      cacheKey = cacheKey,
                                      thumbnailLocalPath = path,
                                      thumbnailReady = true,
                                      previewKind = "thumbnail",
                                      badgeExt = file.extension,
                                      mimeType = file.mimeType,
                                      sourceFileName = file.name,
                                      updatedAt = System.currentTimeMillis(),
                                  )
                              )
                          )
                      }
                  } finally {
                      pendingThumbnailDownloads -= file.messageId
                  }
              }
          }
      }
  }
  ```
- Call `scheduleThumbnailDownloads()` at the end of `refreshFilesInternal()` (after DB operations) and also after any operation that might add new files (e.g., after `upload` completes). For simplicity, we can also call it from `refresh` already covers after upload since upload calls `refreshFilesInternal` in the end. So placing at end of `refreshFilesInternal` suffices.

### 3.5 Ensure imports and error handling
- Wrap `gateway.downloadThumbnail` in try/catch to avoid crashes.
- Use existing `thumbnailSemaphore` (permits=8) to limit concurrent downloads.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Skipping deletion on first refresh may leave stale files if remote is truly empty. | Deletion will happen on subsequent refresh once TDLib is synced. |
| Skipping delete when `files.size == 100` may keep old files if user has >100 and deletes some remotely. | Those stale files will eventually be pruned when the user's total drops below the limit or via manual cleanup later. Acceptable trade-off for stability. |
| Async thumbnail downloads may increase network usage. | Limited by semaphore (8 concurrent) and only for files actually viewed. |
| Changing visibility of `requestDownload`/`awaitDownloadPath` might affect other code. | They are only used within the same file; internal exposure is safe. |
